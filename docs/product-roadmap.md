## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can demo what you've built.
2. **Infrastructure before features.** Phase 0 sets up the monorepo, database, auth, and worker so feature work is fast and clean.
3. **Magic moment first.** The scan → diff → receipt loop (Phases 2–4) is the core value. Billing and polish come after it works end to end.
4. **Test as you go.** Every task's Notes end with a verification step. Severity rules, URL normalization, and diff thresholds get unit tests the moment they exist.
5. **Progressive enhancement.** Simplest working version first (e.g., polling before realtime, textarea editing before rich text), polish later.
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent (e.g. CodeRabbit) review it before starting the next phase. External review catches issues the coding agent won't flag.
7. **Never touch scanned sites destructively.** Scans are read-only GETs. No form submissions, no POSTs to customer sites, concurrency-capped requests. This is a product principle enforced in code.
8. **Severity rules live in one place.** `packages/shared/src/severity.ts` is the single source of truth consumed by worker and UI. Never duplicate classification logic.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the beginning of each phase.
2. **Read selectively:** Each phase lists its Reference sections — the specific parts of the PRD and vision doc needed for that phase. The agent should read only those sections, not the entire documents.
3. **Let it work:** The agent reads the roadmap, finds the first unchecked task, implements it, and marks it complete.
4. **One session = one phase (ideally):** Try to complete a full phase in one session for best continuity. If you need to stop, the agent can resume from the last unchecked task.
5. **Push a PR for review:** When a phase is complete, push the work as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review it before starting the next phase. Address findings on the same branch, re-review, merge only when clean.
6. **Need more context?** If a task references a section not in the phase's Reference sections, the agent should read just that section on demand.

### Session Tips

- **Don't read everything:** The PRD and vision doc are large. Each phase's Reference sections tell the agent exactly what to read. Loading the full documents wastes context.
- **Don't skip tasks:** Tasks are ordered intentionally — the scan engine (Phase 2) is reused by scans (Phase 3), which feed reports (Phase 4). Skipping creates dependency issues.
- **Verify after each task:** Every task's Notes end with a Verify step. Do it before marking the checkbox.
- **Real sites are the test bed:** From Phase 2 on, keep one or two real WordPress sites (or a local WP install you can break on purpose) available for verification. Fixtures test logic; real sites test reality.
- **Severity logic changes go one place:** Any classification/threshold change happens in `packages/shared/src/severity.ts` (+ `visual.ts` constants) with tests, never inline elsewhere.
- **Update the status line:** After completing tasks, update the header: `**Status:** X/89 tasks complete` and `**Current Phase:** Phase N — {Title}`.
- **Founder-action tasks:** Some steps (Supabase/Stripe/Vercel/Railway dashboards, DNS) need the founder. When you hit one, do everything automatable, then hand the founder an explicit numbered checklist and pause that task.

### Branch & PR Convention

- Branch per phase: `phase-{N}/{kebab-case-title}` (e.g. `phase-2/scan-engine-and-baselines`).
- PR title: `Phase {N}: {Phase Title}`. Body: the phase goal, `{X} tasks completed (TASK-{first}–TASK-{last})`, what to verify manually, and the Reference sections used.
- Merge only after review is clean. The next phase builds on merged `main`.

### Prompt Templates

**Starting a new phase:**
> "Read docs/product-roadmap.md and find the current phase. Read only the Reference sections listed for that phase from docs/prd.md and docs/product-vision.md (and docs/design.md if it exists). Start working on the first unchecked task. After completing each task, update the checkbox to [x] and the Status line in the roadmap file. Continue through the phase."

**Resuming after a break:**
> "Read docs/product-roadmap.md. Find where we left off (first unchecked task). Read only the Reference sections listed for the current phase. Continue from the first unchecked task."

**After completing a phase:**
> "Phase [N] is complete. Create a branch called phase-{N}/{slug}, commit all work, push, and open a PR targeting main. Title it 'Phase {N}: {Title}' and include the phase goal and completed task count in the body."

**Fixing an issue:**
> "There's a problem with [description]. Read the relevant section of docs/prd.md for the expected behavior and fix it. Don't mark any new tasks complete until the fix is verified."
